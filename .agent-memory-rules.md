# Agent Memory Rules

**Every agent MUST follow these rules before, during, and after any coding task.**

---

## 🔹 Before Coding — Search Memory

Before writing any code, search memory for relevant context:

```
mem0_search(query="<what am I about to do>", agent_id="shared_rules")
```

Examples:
- `mem0_search("TDD rules for writing tests")`
- `mem0_search("SignalCraft architecture conventions")`
- `mem0_search("how to handle database migrations")`

**Why:** Avoid repeating past mistakes, follow established patterns, respect TDD rules.

---

## 🔹 During Coding — Follow Retrieved Rules

- If TDD rules are found → **strictly follow Red → Green → Refactor**
- If architecture rules are found → **follow the established patterns**
- If coding style rules are found → **match the project conventions**
- If bug fix memories are found → **avoid the same mistakes**

---

## 🔹 After Coding — Update Memory

After completing a task, store learnings:

```
mem0_add(content="<what happened>", agent_id="shared_rules", category="<category>")
```

**What to store:**
| Category | When to store |
|----------|--------------|
| `bugfix` | You fixed a bug — store root cause and solution |
| `architecture` | You made a structural decision — store why |
| `convention` | You discovered a pattern — store it for future |
| `preference` | User expressed a preference — store it |
| `plan` | Implementation plan for incomplete work — store it |

**Examples:**
```
mem0_add("Redis caching layer added to /api/strategies endpoint to reduce DB load", category="architecture")
mem0_add("Bug: Dhan API returns column-oriented data {open: [...], high: [...]}, not row-oriented", category="bugfix")
mem0_add("User prefers snake_case for Python, camelCase for TypeScript", category="convention")
```

---

## 🔹 TDD Rules (Non-Negotiable)

These rules are always active. **Never skip any step:**

1. Always write a failing test (red) before writing any implementation code
2. Write the minimum amount of code to make the failing test pass (green)
3. After green, refactor to improve structure while keeping all tests passing
4. The TDD cycle is strictly: Red → Green → Refactor
5. Never modify a test to make it pass — fix the implementation instead
6. One failing test at a time — never write multiple failing tests simultaneously
7. Test file must exist and test must fail before any implementation file is created

---

## 🔹 Memory Etiquette

- **Be specific** — "Fixed N+1 query in user listing" not "fixed a bug"
- **Include context** — mention the file, endpoint, or feature
- **Store decisions** — why you chose approach A over B
- **Store failures** — what didn't work and why
- **Don't store noise** — no shell dumps, raw diffs, or error logs
